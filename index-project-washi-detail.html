ภาพรวมของงาน

“Washi” คือโปรเจกต์บริการดูแลรถที่ผมสร้างขึ้นเพื่อฝึกออกแบบ UX/UI แบบครบทั้งระบบ โดยออกแบบทั้งหน้าสำหรับลูกค้า (Customer-facing) และแดชบอร์ดสำหรับผู้ดูแลระบบ (Admin Dashboard) เพื่อท้าทายตัวเองในโจทย์ที่มีความเป็น Full-system มากขึ้น ฝั่งลูกค้า ผู้ใช้สามารถเลือกบริการล้างรถ กรอกรายละเอียดรถ จองช่วงเวลาเข้ารับบริการ และติดตามสถานะคำสั่งซื้อได้ ฝั่งแอดมิน ผมออกแบบแดชบอร์ดสำหรับจัดการการจองที่เข้ามา สร้าง/โพสต์โปรโมชัน และติดตามรายได้ภาพรวมของธุรกิจ โปรเจกต์นี้ช่วยให้ผมได้คิดเชิง Business Logic และ User Flow มากขึ้น รวมถึงได้เห็นความแตกต่างของการออกแบบสำหรับผู้ใช้งานทั่วไป เทียบกับการออกแบบสำหรับทีมงานภายในองค์กรอย่างชัดเจน

ปัญหา

1.เจ้าของรถหลายคนมักไม่มีเวลา ไม่มีพื้นที่ หรือมีงบประมาณจำกัดทำให้ดูแลความสะอาดรถได้ไม่ต่อเนื่องหรือไม่คุ้มค่า 2.ร้านคาร์แคร์จำนวนมากยังใช้การจองแบบแมนนวลไม่มีระบบติดตามงาน ส่งผลให้การประสานงานภายในไม่มีประสิทธิภาพ
3.ระบบส่วนใหญ่ยังขาดเครื่องมือฝั่งแอดมินที่ครบถ้วนสำหรับจัดการโปรโมชัน บริการ และดูข้อมูลเชิงวิเคราะห์

แนวทางแก้ไข

สำหรับลูกค้า

1.ขั้นตอนการจองที่ใช้งานง่าย: เลือกบริการ ระบุประเภทรถ ประเภทเชื้อเพลิง สีรถ และช่วงเวลาที่ต้องการ
2.แสดงคิวการให้บริการแบบเรียลไทม์
3.เลือกบริการเสริม และสามารถขอใบเสร็จได้
4.ตรวจสอบประวัติการจองและรีวิวที่ผ่านมา พร้อมติดตามสถานะงานได้

สำหรับAdmin

1.แดชบอร์ดสรุปรายได้รายวัน รายเดือน และรายปี
2.จัดการบริการ ราคา และบริการเสริมได้จากหลังบ้าน
3.ดูคำขอ/รายละเอียดจากลูกค้า และส่งออกรายงานได้
4.สร้างและโพสต์โปรโมชัน รวมถึงดูแล/คัดกรองรีวิวจากลูกค้า

กระบวนการ UX

Step 1 Empathize

จากการทำ Desk Research ผมพบปัญหาหลักของเจ้าของรถในการล้างรถเอง ได้แก่ พื้นที่จำกัด (อยู่คอนโด/ในเมือง), ขาดทักษะและความมั่นใจ, อุปกรณ์ราคาแพงไม่คุ้ม, ไม่อยากล้างเอง, ไม่มีเวลา, และบางคนใช้คาร์แคร์เพื่อ หาที่จอดรถชั่วคราว ด้วย ข้อมูลเหล่านี้ทำให้ Washi ถูกออกแบบมาเพื่อเป็นบริการล้างรถที่ ใช้งานง่าย เข้าถึงสะดวก และเป็นมืออาชีพ ช่วยแก้ทั้งปัญหาในการใช้งานและความรู้สึกของผู้ใช้พร้อมกัน

หลังจากทำการ Persona แล้วจะทำการเอาปัญหาและเป้าหมายของกลุ่มผู้ใช้แต่ละกลุ่มมาตั้งคำถามในการคิดฟีเจอร์ที่จะตอบโจทย์กับระบบงานนี้

How might we

User Insights

ปัญหาที่พบ

ผู้ใช้ส่วนใหญ่มีข้อจำกัดด้านเวลาและพื้นที่ในการล้างรถ โดยเฉพาะผู้ที่อาศัยอยู่ในคอนโดมิเนียมหรือเขตเมือง พวกเขาไม่สะดวกที่จะล้างรถเอง ไม่ต้องการลงทุนในอุปกรณ์ราคาแพง และไม่ชอบรอคิวนาน

ความต้องการของผู้ใช้

  • บริการล้างรถที่สะดวกสบาย สามารถจองล่วงหน้าได้ง่าย
  • ราคาและระยะเวลาการให้บริการที่ชัดเจนก่อนยืนยันการจอง
  • แพลตฟอร์มที่น่าเชื่อถือพร้อมรีวิวจากผู้ใช้จริง
  • โซลูชันที่เรียบง่าย ประหยัดเวลา และไม่ยุ่งยาก
  • ความรู้สึกควบคุมได้โดยไม่รู้สึกว่าถูกกดดัน

เราจะทำอย่างไรได้บ้าง

  • เราจะทำให้การล้างรถเป็นเรื่องง่ายและเข้าถึงได้สำหรับผู้ที่มีเวลาหรือพื้นที่จำกัดได้อย่างไร?
  • เราจะช่วยให้ผู้ใช้ไว้วางใจบริการผ่านรีวิวและข้อมูลบริการที่โปร่งใสได้อย่างไร?

Admin Insights

ปัญหาที่พบ

เจ้าของธุรกิจเผชิญกับความท้าทายในการจัดการการจองและข้อมูลของลูกค้า ปัญหาต่างๆ เช่น การจองซ้ำซ้อน ข้อมูลลูกค้าที่กระจัดกระจาย และการขาดข้อเสนอแนะที่นำไปใช้ได้ ทำให้ยากต่อการปรับปรุงบริการหรือดำเนินงานอย่างมีประสิทธิภาพ

ความต้องการของผู้ใช้

  • ระบบที่ช่วยจัดการการจองของลูกค้าได้อย่างง่ายดายและหลีกเลี่ยงการซ้ำซ้อน
  • รายงานรายวันและการมองเห็นประสิทธิภาพการจอง
  • วิธีการรับและจัดการข้อเสนอแนะของลูกค้า
  • ความสามารถในการอัปเดตโปรโมชั่นและบริการได้อย่างอิสระ
  • ฐานข้อมูลส่วนกลางของข้อมูลลูกค้าและประวัติการจอง

เราจะทำอย่างไรได้บ้าง

  • เราจะช่วยผู้ดูแลระบบจัดระเบียบและจัดการการจองได้อย่างมีประสิทธิภาพมากขึ้นได้อย่างไร?
  • เราจะช่วยผู้ดูแลระบบรวบรวมและนำข้อเสนอแนะของลูกค้าไปใช้เพื่อปรับปรุงบริการได้อย่างไร?
  • เราจะช่วยให้ผู้ดูแลระบบอัปเดตเนื้อหา (เช่น บริการ โปรโมชั่น) โดยไม่ต้องพึ่งพาผู้พัฒนาได้อย่างไร?

Step 3 Ideate

ในช่วงคิดไอเดีย ผมจะคิดฟีเจอร์จากปัญหาที่พบในการรีเสิร์ชผู้ใช้ แล้วนำมาจัดหมวดและจัดลำดับความสำคัญด้วยตาราง ผลกระทบ/ความพยายาม (Impact/Effort) เพื่อโฟกัสฟีเจอร์ที่สำคัญที่สุดต่อทั้งฝั่ง ผู้ใช้ และ แอดมิน

จัดลำดับความสำคัญฟีเจอร์ (Impact–Effort Matrix)

เป้าหมายฝั่งผู้ใช้

ทำให้บริการทำความสะอาดรถ เข้าถึงง่ายขึ้น ประหยัดเวลา และน่าเชื่อถือขึ้น

Easy Wins (ผลกระทบสูง–ทำง่าย):

  • เห็นราคาก่อนจอง
  • เลือกประเภทบริการ
  • จองล่วงหน้า เลือกวัน-เวลา
  • รีวิวบริการ
  • ใบเสร็จ/ยืนยันบริการ
  • ติดตามสถานะการจอง

Big Bets (ผลกระทบสูง–ทำยาก):

  • ระบบจ่ายเงินออนไลน์
  • แสดงข้อมูลช่างผู้ให้บริการ
  • ระบบแต้ม/รางวัล

จัดลำดับความสำคัญฟีเจอร์ (Impact–Effort Matrix)

เป้าหมายฝั่งแอดมิน

มีเครื่องมือให้แอดมินจัดการ การจอง บริการ และข้อมูลลูกค้า ได้ง่ายขึ้น และพัฒนาการทำงานของธุรกิจด้วย ข้อมูลเชิงลึก + ฟีดแบ็ก

Easy Wins (ผลกระทบสูง–ทำง่าย):

  • จัดการการจองรายวัน
  • ติดตามความพร้อมของพนักงานและสถานะงาน
  • รายงานรายวันและรีวิวลูกค้า
  • แก้ไขบริการ / ออกใบเสร็จ

Big Bets (ผลกระทบสูง–ทำยาก):

  • จัดการข้อมูลลูกค้าและพนักงาน
  • สร้างโปรโมชัน/ส่วนลด
  • โทรหาลูกค้าจากในระบบได้เลย

Step 4 Prototype

Information Architecture flow

User Information Architecture

หลังจากคัดเลือกและจัดลำดับฟีเจอร์แล้ว ขั้นต่อไปคือแปลงฟีเจอร์เหล่านั้นเป็น โฟลว์ IA เพื่อให้เห็นภาพว่าแต่ละฟีเจอร์จะถูกจัดวางในระบบอย่างไร และผู้ใช้จะเดินทางใช้งานในแพลตฟอร์มแบบไหน

เป้าหมาย IA ฝั่งผู้ใช้

ชัดเจน : ผู้ใช้เข้าใจบริการและใช้งานได้ง่าย ไม่สับสน
สะดวก : ลดขั้นตอนที่ติดขัดในการจอง จ่ายเงิน และรีวิว
น่าเชื่อถือ : แสดงรายละเอียดบริการ ราคา และข้อมูลช่างอย่างโปร่งใส

ตัวอย่างหมวดเมนูของ IA User:

  • Home : ภาพรวม/ส่วนหัวหลักและปุ่มเรียกให้ทำงาน
  • Services :รายการบริการ
  • Booking :เลือกบริการ เวลา สถานที่ ข้อมูลรถ และวิธีชำระเงิน
  • Order Summary / Confirmation :สรุปรายละเอียดการจอง + ใบเสร็จ/การยืนยัน
  • My Bookings :สถานะและประวัติการจอง
  • Reviews :อ่าน/เขียนรีวิวบริการ
  • Profile :แก้ไขข้อมูลส่วนตัว ดูแต้มสะสม และตั้งค่า

Admin Information Architecture

ตัวอย่าง Flow ของฝั่ง Admin

เป้าหมาย IA ฝั่งแอดมิน

ทำงานไว : จัดการการจองและบริการได้ลื่นไหล
รวมข้อมูลไว้ที่เดียว : จัดระเบียบข้อมูลลูกค้าและช่างให้เข้าถึงง่าย
มีข้อมูลเชิงลึก : ดูรายงานและฟีดแบ็กเพื่อพัฒนาคุณภาพบริการ

ตัวอย่างหมวดเมนูของ IA Admin:

  • Dashboard : ภาพรวมการจองวันนี้ + สถิติสำคัญ
  • Bookings Management :ดู/แก้ไข/มอบหมายงานนัดหมาย
  • Customer Data :โปรไฟล์ลูกค้าและประวัติการใช้บริการ
  • Technician Availability :เช็คความพร้อมช่าง มอบหมายงาน ติดตามสถานะ
  • Services & Pricing Editor :ปรับบริการและราคาได้ตลอด
  • Promotion Manager :สร้างและตั้งเวลาโปรโมชัน
  • Daily Reports :รายรับ ประวัติการจอง และติดตามฟีดแบ็ก
  • Settings :จัดการข้อมูลธุรกิจ ช่องทางติดต่อ และข้อมูลล็อกอิน

  • เลย์เอาต์: ใช้ Grid-based layout เช่น Hero 2 คอลัมน์ และการ์ดฟีเจอร์/รีวิว 3 คอลัมน์ รองรับมือถือโดยยุบเป็น 1 คอลัมน์ และเมนูเปลี่ยนเป็น Hamburger
  • คอมโพเนนต์: สไตล์คงที่ เช่นการ์ดมุมโค้ง และดีไซน์ปุ่มที่ไปในทิศทางเดียวกัน
  • ประสบการณ์ผู้ใช้ (UX): เน้นปุ่ม CTA ที่ชัด และการไหลของข้อมูลเป็นเหตุเป็นผล ทำให้ใช้งานแล้วเข้าใจง่ายและดูสวยน่าใช้

Design System

ผมออกแบบให้ สม่ำเสมอและ Responsive ใช้งานได้ดีทั้งเดสก์ท็อป แท็บเล็ต และมือถือ

  • ฟอนต์ : ใช้ Noto Sans Thai UI ทุกหน้าจอ เพื่อความอ่านง่ายและดูทันสมัย
  • Typography :ปรับสเกลตามอุปกรณ์ เช่น H1 36px บนเดสก์ท็อป / 24px บนมือถือ
  • ชุดสี :Primary: #3588FC Secondary: #41528C Accent/ปุ่ม: #0D947F
  • ปุ่ม :มีทั้ง Primary/Secondary รองรับทุกอุปกรณ์ พร้อมสถานะ Default / Hover / Focused / Disabled
  • ช่องกรอกข้อมูล :มีป้ายกำกับชัดเจน พร้อมสถานะ Focus / Error / Disabled และปรับขนาดให้เหมาะกับแต่ละอุปกรณ์
  • เลย์เอาต์ :ความกว้างสูงสุด 1296px (เดสก์ท็อป) ปรับ padding อัตโนมัติบนแท็บเล็ต/มือถือ
